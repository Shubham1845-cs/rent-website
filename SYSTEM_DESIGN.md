# System Design Write-Up

## Overview
Rent & Flatmate Finder is built using the MERN stack (MongoDB, Express, React, Node.js). The architecture is designed to cleanly separate concerns while providing real-time capabilities and AI-driven matching. The core business logic revolves around four primary features: Compatibility Scoring, LLM Fallback Mechanisms, Real-Time Chat, and Email Notifications.

## 1. Compatibility Scoring Design
The compatibility scoring engine is designed to minimize real-time computational overhead while maximizing the user experience. Instead of running heavy computations on every search query, scores are calculated asynchronously and cached in a dedicated `CompatibilityScore` MongoDB collection.

When a tenant queries for listings, the system first checks the `CompatibilityScore` collection using a compound index on `(tenantId, listingId)`. If a cached score exists, it is instantly joined with the listing data. If a score is missing (e.g., for newly posted listings), the system triggers the scoring service to compute it on the fly, caches the result in the database, and then returns it. This design ensures that browsing listings is fast (O(1) lookup per listing after the initial cache) and prevents unnecessary API calls to the LLM provider.

When an owner updates a listing (e.g., changes the rent or available date), the system invalidates the cached scores for that specific `listingId` by issuing a bulk delete command on the `CompatibilityScore` collection. The scores are seamlessly recomputed the next time a tenant views that listing.

## 2. LLM Integration and Fallback
To provide intelligent, natural-language explanations for why a listing is a good or poor match, the platform integrates with Google Gemini (`gemini-2.0-flash`). The LLM evaluates a strict prompt containing the tenant's profile constraints (budget, location, move-in date) against the listing's specifications and returns a JSON object with a numeric `score` (0-100) and a brief string `explanation`.

**Graceful Degradation (Fallback Mechanism)**
Relying strictly on third-party APIs introduces points of failure (e.g., rate limits, network timeouts, or service outages). To guarantee continuous uptime, the `scoringService` employs a try-catch mechanism with a strict 10-second timeout using an `AbortController`. 

If the LLM call fails, times out, or returns improperly formatted JSON, the system immediately degrades to a pure mathematical **Rule-Based Fallback Scorer**. 
- **Budget (50%):** Perfect score if rent is within bounds; linearly degrades as rent exceeds bounds.
- **Location (30%):** Perfect score for exact multi-word matches; partial score for single-word substring matches.
- **Move-In Date (20%):** Tiered scoring based on absolute day differences (e.g., 20 points for ≤7 days).

The fallback logic executes synchronously and ensures the user always receives a score, guaranteeing the core business flow is never interrupted by AI downtime. The `method` field in the database (`'llm'` vs `'fallback'`) records how the score was derived for analytics.

## 3. Real-Time Chat Implementation
Once an owner accepts a tenant's interest request, real-time communication is unlocked. This is implemented using **Socket.io** over WebSockets, bypassing traditional HTTP polling.

- **Room Architecture:** Chat sessions are completely private. When a user navigates to a chat page, the frontend emits a `join_room` event. The room ID is deterministically generated by concatenating the `requestId` and `userId` securely, ensuring only authorized participants can join.
- **Message Persistence:** Before a message is broadcasted to the room via `io.to(roomId).emit()`, it is synchronously saved to the `ChatMessage` MongoDB collection. This guarantees that no messages are lost if a user's connection drops, and historical messages can be loaded via a standard REST API (`GET /api/chat/:requestId/messages`) when the chat page mounts.
- **Security:** The Socket.io connection utilizes a middleware that validates the user's JWT exactly like the REST API. If the token is invalid or missing, the socket connection is rejected outright.

## 4. Notification Flow
To keep users engaged without requiring them to actively refresh the dashboard, the system utilizes asynchronous, fire-and-forget email notifications built on top of **Nodemailer**.

- **High-Score Alerts:** When a tenant expresses interest (`POST /api/interest`), the backend evaluates the cached compatibility score. If the score is >80, it triggers an asynchronous email to the owner. This occurs outside the main request thread, allowing the HTTP response to return immediately without waiting for the SMTP server.
- **Status Updates:** When an owner patches an interest request (`PATCH /api/interest/:id`) to `accepted` or `declined`, the backend determines the tenant's email and dispatches the corresponding notification. The `accepted` email contains a deep link directly to the newly unlocked Chat interface to drive engagement.
- **Error Boundaries:** All email dispatches are wrapped in isolated `try-catch` blocks. If the SMTP server is unreachable or the email fails to send, the error is logged to the console, but the core database transaction (e.g., accepting the request) still succeeds.
